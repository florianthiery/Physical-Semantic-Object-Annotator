Datenmodel

Das Semantic Object Data Model (SOD) zeigt ein allgemeingültiges Modell zur Annotation semantischer Objekte mit Hilfe der Technologie des Semantic Web, dem Resource Description Framework (RDF). Als semantisches Objekt wird eine HTTP Resource gesehen, welche ein reales (physisches) Objekt der Realität beschreibt. Beispiele hierfür sind z.B. Personen, Türen, Schränke, Computer, Kirchen, etc. Zur Generierung semantischer Objekte kann der Semantic Object Service genutzt werden. Die Annotation folgt dem Open Annotation Data Model. Eine Annotation ist hierbei mit einer Menge von Resourcen über einen Body und Target verbunden. Hierbei sagt der Body etwas über das Target aus. Der verbindende Knoten, die Annotation, enthält Metadaten wie das Datum, den Annotator, sowie den Typ (bzw. die Motivation). Schlieﬂen sich mehrere Resourcen an den Body an, ist ein Graph zu erzeugen, z.B. ein nicht im Web adressierbarer leerer Knoten (blank Node). Die Annotation kann hier als beschreibenes Element gesehen werden, das heiﬂt ein Annotator erzeugt Attribute wie Name, Typ, geometrische Repräsentation des semantischen Objekts. Darüber hinaus können über eigene Prädikate und anzuschlieﬂende Resourcen weitere Eigenschaften, wie z.B. eine Farbe benannt werden. Beziehungen zwischen zwei semantischen Objekten werden über einen zweiten Annotationstypen beschrieben. Ein semantisches Objekt kann zudem mehr als eine Annotation von einer oder mehreren Personen enthalten.


Quelle: Open Annotation Data Model

Ein zu annotierendes semantisches Objekt enthält verschiedene Klassendefinitionen, allen voran dctypes:PhysicalObject. Dies spiegelt die Idee der Annotation real existierender physischer Objekte wieder. Weitere Typdefinitionen sind zur Verlinkung des semantischen Objekts mit Hilfe von RDF mit den gängigsten Definitionen vorgesehen. Die Open Annotation Annotation zeigt einen Body auf, der über das Target etwas aussagt. Im Fall des Semantic Objekt Data Model kann der Body und das Target als identisch angesehen werden, welches lediglich über eine Annotation verbunden ist, so dass die annotierten Eigenschaften eindeutig dem Ersteller zugeordnet werden können. Jede Annotation enthält gemäﬂ Open Annotation Model eine Klassendefinition, eine Open Annotation Motivation, das Datum der Erstellung nach ISO 8601, sowie den Annotator. Dieser wird mit Hilfe des Fried of a Friend (FOAF) Modells modelliert. Der Body der Annotation besteht aus einem oder mehreren Graphen (hier leere Knoten, blank nodes). Hierbei ist zu beachten, dass ein primärer leerer Knoten mit einer textlichen Beschreibung des semantischen Objekts (prefLabel, altLabel, scopeNote), einem Labeling (skos:Concept) zur Typdefinition und einer geometrischen Repräsentation (Feature nach GeoSPARQL Geometrie) vorhanden sein muss, wobei die Prädikate festgelegt sind. Das präferierte Label ist als nicht optional anzusehen. Jedes weitere Prädikat und damit verbundene Resourcen ist an einen eigenen leeren Knoten zu koppeln. Diese leere Knoten enthalten gängige Klassendefinitionen, sodass übliche RDF Prädikate genutzt werden können. Der nachfolgende Graph zeigt diesen Sachverhalt auf und wird durch die verwendeten Namespaces und ein Beispiel in Turtle Syntax ergänzt.

@prefix dctypes: <http://purl.org/dc/dcmitype/> .
@prefix skos: <http://www.w3.org/2004/02/skos/core#> .
@prefix trig: <http://www.w3.org/2004/03/trix/rdfg-1/> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix geo: <http://www.opengis.net/ont/geosparql#> .
@prefix semobj: <http://semanticobjects.org/> .
@prefix agent: <http://agents.org/> .
@prefix anno: <http://annotations.org/> .
@prefix label: <http://labelconcepts.org/> .
@prefix feature: <http://features.org> .

#Annotation
[anno]
  a oa:Annotation ;
  oa:motivatedBy oa:describing ;
  oa:annotatedAt "[dateFormat ISO 8601]"^^xsd:dateTime ;
  oa:annotatedBy [agent] ;
  oa:hasTarget [semanticobject] ;
  #Body default
  oa:hasBody [
    a trig:Graph, skos:Concept ;
    #Class Definition
  skos:exactMatch [labelconcept] ;
    #Feature
  skos:relatedMatch [feature] ;
  #Labeling
  skos:prefLabel "[string]" ;
  skos:altLabel "[string]" ;
  skos:scopeNote "[string]"
  ] , [ #additional body
    a trig:Graph, skos:Concept, rdfs:Resource, owl:Thing ;
  [rdf:property] "[string]" ;
  ] .
[semanticobject] a dctypes:PhysicalObject, skos:Concept, rdfs:Resource, owl:Thing .
[agent] a foaf:Person .
[labelconcept] a skos:Concept .
[feature] a geo:SpatialObject, geo:Feature, skos:Concept .

Ein Beispiel

Ein real existierendes physisches Objekt soll annotiert werden. Hierbei handelt es sich um die Eingangstür des Büro Gamma in den Räumen des Instituts für Raumbezogene Informations- und Messtechnik (i3mainz). Der Annotator ist Florian Thiery, welcher durch seinen Namen und eine Emailadresse eindeutig identifiert ist. Die Annotation selbst wird durch den Annotator und die Uhrzeit einzigartig. Es handelt sich lediglich um beschreibende Informationen, somit ist die Motivation als describing festzulegen. Im Hauptknoten ist zunächst das Objekt textlich zu beschreiben, welches durch diverese prefLabel, altLabel und scopeNotes ermöglicht wird. Als Typ wird eine Tür festgelegt, welche in einem SKOS Concept definiert ist und Links zu beschreibenden Resourcen enthält. Die Gammatür ist ebenfalls als Geometrie vorhanden (hier aus dem IBR Generic Viewer). Als weitere beschreibende Eigenschaft ist ein Wikipedia Artikel anzusehen. Dieser wird über ein Prädikat mit einem neuen Nebenknoten mit der Annotation verbunden.

Multiple Annotationen

& Semantic Object Relationen

Ein semantisches Objekt kann mehrere Annotation enthalten. Diese können von einer Person, wie auch von mehreren Personen stammen. Hierdurch ist beispielweise ein wissenschaftlicher Diskurs möglich. Das verbindende Prädikat hierbei ist das Target. Durch die festgelegte Struktur ist hierbei ein gleiches Schema bei einer Abfrage möglich. Soll eine Relation zwischen zwei semantischen Objekten erzeugt werden, wird eine Body-lose Annotation erstellt. Das Target der Annotation enthält hierbei ein Prädikat zu einem weiteren semantischen Objekt. Das Prädikat ist in beiden Richtungen gleich zu wählen. Durch diese Struktur ist der Urheber dieser Relation weiterhin erkennbar, eine semantische Abfrage der Realtionen ist jedoch um einiges vereinfacht.
